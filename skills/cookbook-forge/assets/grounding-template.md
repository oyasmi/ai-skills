# 接地事实与写作规范（内部文档 · 非成书内容）

> **本文件是本食谱全体写作的唯一事实源。** 正文里关于 <主题> 的任何论断，都必须能在本文件中找到依据，或来自已核实的信源。**严禁凭训练记忆臆测细节。** 不确定的，宁可标注「（待核实）」也不要编造。把它当作正文每一句话的「弹药库」——写到一句无法在此追溯的话时，就该停下去补研究，而不是继续写。
>
> 用法：研究阶段把事实逐条记入下面的「事实清单」；写作时只引用清单里的条目；成书的「信源索引」附录由「信源台账」一节生成。删除本说明与示例行后再使用。

### 读者画像（动笔前必填 —— 决定深度的准星）

| 项 | 内容 |
|---|---|
| 角色 | <读者是谁：新维护者？从 X 迁移来的工程师？> |
| 入口知识 | <他必然已经懂的 → 决定起点放多低> |
| 必然不懂 | <他必然缺的前置 → 每章要不要补> |
| 出口能力 | <读完能独立做成的那一件事> |

> 「错误心智模型」一栏是精品与合格的分水岭。只要能定位到读者一个具体的错误理解，就为一（几）章找到了真正的钩子——那种章存在的意义就是纠正它（例：读者以为「信息收敛 = 丢信息 = 坏」，而真相是「有损压缩本身带来价值」）。填不出这一栏，说明对读者还不够了解，回去补研究，别硬写。

### 证据分级 —— 两个正交维度

一条事实同时带「信源强度」和「所处层面」两个标签，二者独立（一条事实可以是 L1-impl 却只有 L3-behavior）：

**① 信源强度（`evidence` 列）**
- **L1 一手 / 已核实** —— 官方文档、规范、源码，或我亲自运行 / 复现过。可直接断言。
- **L2 二手 / 转述** —— 第三方声称、未能复现。正文必须显式标注「据 X」「某文称」，**不得**洗成事实。
- **L3 推断 / 前瞻** —— 我的推理或预测。正文标「这暗示……」「截至 <日期>……」。

**② 所处层面（`层面` 列）—— 尤其针对代码库考古**
- **spec 意图** —— 设计文档 / PRD / 架构稿说「应当如此」。
- **impl 现实** —— 源码说「实际如此」。
- **behavior 实测** —— 我真跑过、亲眼看到的运行时行为。

> **设计文档 ≠ 源码。** 设计稿写的是意图，源码写的是现实，二者经常背离。**冲突时正文一律以源码为准，并把冲突本身写进书里**（「设计文档称 X，但 master @abc123 实际是 Y」——这种落差往往正是一章里最有价值的一节）。
>
> **硬规矩：** 凡是关于本代码库的 **L1 断言**，必须在「信源 / 出处」列带上 `file:line` 或 `commit`；给不出精确出处的，一律降级为 L2。
