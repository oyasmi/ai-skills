## 选择工作模式

先确定这次 turn 要产生什么，不要把调查、方案评审和实现授权混在含糊的一句话里。

| 模式 | 使用条件 | 默认权限 | 预期输出 |
| --- | --- | --- | --- |
| 调查 | 原因未知，需要收集证据 | 只读；允许运行非破坏性诊断 | 根因、证据、影响范围、建议下一步 |
| 规划 | 需求复杂、边界不清或改动风险较高 | 只读；不要修改实现 | 方案、影响文件、取舍、风险、验证计划 |
| 实现 | 目标和边界清楚，可以直接交付 | 允许范围内的本地编辑和验证 | 变更、验证证据、剩余风险 |
| 审查 | 已有稳定 diff 或产物需要独立检查 | 只读；不要顺手修改 | 影响正确性、需求或安全性的具体缺口 |
| 修复审查意见 | 已有编号发现需要落实 | 只修改发现覆盖的范围 | 每项处理结果和重新验证证据 |

仅当专业视角会改变判断标准时指定角色，例如安全审查或数据库迁移审查。不要用“你是一名优秀工程师”一类无可验证作用的人设填充上下文。

## 选择有效上下文

优先提供能改变结果的信息：

1. 当前工作目录以及适用的 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md` 或项目规范路径。
2. 精确的文件路径、符号名、接口、配置键或提交范围。
3. 最小复现步骤、输入、期望输出、实际输出和完整错误文本。
4. 已经确定的设计决策、兼容性要求和必须保持的行为。
5. 权威规格、issue、设计稿、测试或 fixture 的路径。
6. 可执行的构建、测试、lint、类型检查或视觉验证方法。

不要提供：

- 与当前目标无关的完整会话历史。
- 整份仓库文件清单或可由外部 Agent 快速发现的常识。
- 没有说明用途的大段日志或文档。
- 已失效的猜测、互相冲突但未标注优先级的要求。
- 密钥、令牌、私钥或生产凭据。

较长的规格、日志或评审意见应写入文件。任务指令中给出绝对路径，并说明从文件中提取什么。不要把同一份内容既完整粘贴又要求读取文件。

如果上下文存在冲突，明确事实来源的优先级。无法判断且选择会显著改变结果时，要求外部 Agent 停止并报告一个最小阻塞问题。

## 首次任务指令模板

### 调查问题，不修改代码

```text
工作模式：调查，不修改代码。

目标：确定 <现象> 的根因，并用可复查的证据解释。

上下文：
- 复现：<命令或步骤>
- 实际结果：<错误或异常行为>
- 重点检查：<路径、符号或日志>

范围：不要修改文件、提交代码或执行外部写入。可以运行非破坏性诊断命令。

交付：报告根因、证据位置、受影响范围和最小修复建议；区分确认事实与推测。无法确认时说明还缺什么证据。
```

### 复杂任务，先规划

```text
工作模式：规划，暂不修改代码。

目标：为 <期望结果> 制定可实施方案。

上下文：
- 先阅读当前目录适用的仓库指令。
- 需求或规格：<路径>
- 相关模块：<路径或符号>
- 已确认决策：<约束>

请调查现状并给出：建议方案、预计修改的文件和接口、关键取舍、兼容性或迁移风险、测试策略，以及仍需确认的问题。不要开始实现。
```

审查方案后，用追加指令明确授权实现，并把已确认的选择转成完成标准。不要只回复“按计划做”。

### 实现边界清楚的需求

```text
工作模式：实现。

目标：<可观察的最终行为>。

上下文：
- 先阅读当前目录适用的仓库指令。
- 相关实现：<路径或符号>
- 当前行为或复现：<证据>
- 参考模式或事实来源：<路径>

范围与边界：
- 允许修改：<范围>
- 必须保持：<兼容性或既有行为>
- 不在范围：<非目标>
- 未经授权，不提交、推送、部署、删除外部资源或扩大范围。

完成标准：
- <行为条件一>
- <边界或回归条件>
- 运行 `<验证命令>` 并通过。

在范围内完成实现、测试和自查。若被阻塞，报告证据和所需的最小补充，不要猜测。最后报告修改文件、检查命令及结果、未验证内容和剩余风险。
```

### 独立审查稳定 diff

```text
工作模式：独立审查，不修改文件。

目标：审查 <diff、commit 或路径> 是否满足以下完成标准：
- <标准一>
- <标准二>

重点检查正确性、回归、边界条件、安全性和测试缺口，并参考 <仓库指令或规格>。只报告会影响明确需求或真实风险的问题，不报告纯风格偏好。

每个发现给出：严重程度、文件和位置、触发条件、影响、证据以及最小修复方向。没有发现时明确说明审查范围和已执行的验证。
```

## 追加与纠偏模板

### 根据失败证据纠偏

```text
观察到：运行 `<命令>` 后出现 <关键失败输出>，与完成标准 <编号或内容> 不符。
需要修正：<具体预期行为>。
保持不变：<已经正确的行为、接口或文件>。
重新验证：先运行 `<聚焦检查>`，再运行 `<必要回归检查>`。

只修复这个偏差；完成后报告根因、修改位置和两项检查的实际结果。
```

### 补充新信息

```text
补充上下文：<新日志、规格路径或已确认决策>。
这会改变：<原计划或实现中的具体部分>。
其余目标、范围和完成标准保持不变。
请据此继续，并重新验证 <检查项>。
```

### 回答外部 Agent 的阻塞问题

```text
关于你提出的 <问题>：采用 <明确选择或事实>。
依据：<规格、路径或用户决定>。
保持原任务的其他边界和完成标准不变。继续完成并报告验证证据。
```

### 批量处理审查意见

```text
工作模式：修复审查意见。

处理以下确认的问题：
1. <发现及位置>
2. <发现及位置>

不处理：<被拒绝或延期的意见及原因>。
保持不变：<非目标和正确行为>。

逐项修复并运行 <验证命令>。最后按编号报告处理结果；无法完成的项提供阻塞证据，不要用无关重构替代。
```

### 放弃污染的会话后重新开始

不要把旧会话的所有往返原样复制给新实例。只保留：

- 原始目标和仍有效的边界。
- 已确认的事实和复现证据。
- 已尝试但失败的方法及其失败原因。
- 当前文件状态或稳定 commit/diff。
- 更新后的完成标准和验证命令。

把这些内容重新组织成一条完整的首次任务指令。

## 验证与交付证据

优先给外部 Agent 可运行的闭环：测试、构建、lint、类型检查、fixture 比对、截图或明确的复现脚本。不要只写“确保没问题”。

让外部 Agent 报告：

1. 实际修改的文件和关键行为。
2. 实际运行的命令，而不是建议以后运行的命令。
3. 每个检查的结果；失败、跳过或无法运行时说明原因。
4. 没有覆盖的环境、平台或边界条件。
5. 仍存在的假设、风险和阻塞项。

不要要求冗长的内部思考过程。要求结论、可复查证据和关键取舍即可。

把外部 Agent 的自验视为第一层证据。编排 Agent 仍要直接检查文件和 diff，并亲自运行与风险相称的验证。视觉任务应检查实际渲染或截图，不能只接受“已匹配设计”的文字声明。

## 会话复用和重新开始

复用同一实例的条件：

- 仍在解决同一个目标。
- 既有决策和已读文件仍有价值。
- 追加内容是新证据、明确选择或具体纠偏。

使用新实例的条件：

- 任务与当前目标无关。
- 原任务范围已经发生实质变化。
- 同一问题连续两次纠偏仍失败，旧上下文充满失败路径。
- 需要不受实现过程影响的独立审查。
- 原实例状态为 `lost` 且无法安全恢复。

创建新实例时使用新的描述性名称。不要为了清理上下文就复用名称并误以为历史已经消失；先确认对应 harness 的会话语义。

## 并行与独立审查

把运行实例隔离和文件系统隔离分开处理：

- 多个写入者使用独立 worktree 和不同 `--cwd`。
- 明确每个分片的输入、输出和文件所有权。
- 避免多个实例共享 Git index、生成目录、数据库或同一测试环境。
- 汇总前先让每个分片完成自己的验证，再在集成工作树运行整体检查。
- 审查者基于稳定 commit 或 diff 工作，不要审查持续变化的目标。

写入者/审查者模式中，不要把实现者的完整推理过程交给审查者。提供原始任务、完成标准、稳定 diff 和必要上下文，让审查者独立判断。把确认后的发现集中交回写入者，避免逐条触发多轮零散修改。

## 常见反例

| 不良任务指令 | 问题 | 改写方向 |
| --- | --- | --- |
| `继续` | 下一步不唯一 | 指明未完成项、保持项和验证方法 |
| `修好它` | 缺少现象和期望 | 给出失败证据、预期行为和范围 |
| `把测试都修了` | 可能掩盖产品缺陷或改坏测试 | 指明失败测试，要求修根因并保持测试意图 |
| `优化一下代码` | 没有可判断的目标和停止条件 | 指明性能、可读性或结构指标及非目标 |
| `检查有没有问题` | 审查范围和发现标准无限 | 指明 diff、完成标准和只需报告的风险类型 |
| `按你的方案做` | 方案可能仍有未决选择 | 明确批准的方案版本、选择和实现授权 |
| `不要改错任何东西` | 无法执行或验证 | 写出必须保持的具体接口、行为和文件范围 |

避免把所有最佳实践都塞进每条任务指令。任务越清楚，模板越应该缩短；只有真实风险需要边界，只有可改变结果的信息值得占用上下文。
